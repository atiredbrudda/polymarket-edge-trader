  <behavior>
    Module follows established pure-function, duck-typed pattern. All calculations use Decimal.
    No SQLAlchemy imports. Scoring functions accept pre-computed data, not ORM objects.

    **ExpertiseScoreResult frozen dataclass:**
    - raw_score: Decimal (weighted composite, 0-100)
    - percentile_rank: Decimal | None (population-relative rank, 0-100; None if not yet computed)
    - win_rate_component: Decimal (scaled 0-100, before weighting)
    - concentration_component: Decimal (scaled 0-100, before weighting)
    - recency_component: Decimal (scaled 0-100, before weighting)
    - sample_size_component: Decimal (scaled 0-100, before weighting)
    - consistency_multiplier: Decimal (1.0-1.1 range; bonus-only, never below 1.0)
    - specialization_label: str ("specialist/specialist", "specialist/generalist", "generalist/specialist", "generalist/generalist")
    - game_slug: str
    - trader_address: str
    - resolved_market_count: int (number of resolved markets used in scoring)

    **Constants:**
    ```python
    DEFAULT_WEIGHTS = {
        "win_rate": Decimal("0.40"),
        "concentration": Decimal("0.25"),
        "recency": Decimal("0.20"),
        "sample_size": Decimal("0.15"),
    }
    MIN_RESOLVED_MARKETS = 5
    RECENCY_HALF_LIFE_DAYS = 90
    ```

    **Functions:**

    1. `calculate_recency_weight(last_resolved_timestamp: datetime, now: datetime, half_life_days: int = 90) -> Decimal`
       - Exponential decay: weight = 0.5 ^ (days_since / half_life_days)
       - Returns Decimal("1.0") if same day or future timestamp
       - At half_life_days: returns ~0.5
       - At 2x half_life: returns ~0.25
       - Uses math.log/math.exp internally, converts to Decimal for output

       DESIGN NOTE (SCOR-04 — recency weighting): Recency is implemented as a separate
       weighted component (~20%) in the composite score, NOT as a modifier on win rate
       itself. This satisfies SCOR-04 ("System applies recency weighting so recent
       performance counts more than old") because: recently active traders get a high
       recency component (near 100), inactive traders get a low recency component (near 0),
       which directly makes "recent performance count more." Applying recency weighting
       TO win rate would be a different semantic (position-level time-weighting of individual
       outcomes) which is not what the requirement asks for.

    2. `calculate_sample_size_confidence(resolved_market_count: int, min_threshold: int = 5, full_confidence_threshold: int = 30) -> Decimal`
       - Returns Decimal("0") if below min_threshold
       - Returns Decimal("1.0") if >= full_confidence_threshold
       - Between: exponential growth curve 1 - exp(-k * (n - min_threshold))
       - k tuned so confidence ~0.95 at full_confidence_threshold

    3. `calculate_expertise_score(positions: list[Any], trader_address: str, game_slug: str, esports_concentration: Decimal, game_concentration: Decimal, consistency_score: Decimal, consistency_signal: str, weights: dict[str, Decimal] = None, now: datetime | None = None) -> ExpertiseScoreResult | None`
       - Returns None if fewer than MIN_RESOLVED_MARKETS resolved positions (excluding void)
       - Components computed:
         a. Win rate: from calculate_win_rate(resolved_positions) — already 0-100 scale
         b. Concentration: game_concentration * 100 (convert 0-1 to 0-100)
         c. Recency: calculate_recency_weight(last_resolved_timestamp, now) * 100
         d. Sample size: calculate_sample_size_confidence(count) * 100
       - Raw score = weighted sum of components (weights default to DEFAULT_WEIGHTS, must sum to 1.0)
       - Consistency multiplier applied AFTER weighted sum (bonus-only per user decision — "multiplier/bonus" means reward for consistency, NOT penalty for inconsistency):
         * consistency_score >= 80 AND consistency_signal == "stable" -> 1.05x (bonus)
         * All other cases -> 1.0x (no penalty, baseline)
         NOTE: The user explicitly said "multiplier/bonus" which implies additive advantage for consistent traders. Streaky/neutral traders receive 1.0x (no effect), never below 1.0x.
       - Clamp final raw_score to [0, 100]
       - percentile_rank set to None (computed in batch via normalize_scores_to_percentiles)
       - specialization_label from classify_specialization(esports_concentration, game_concentration, game_slug)

    4. `normalize_scores_to_percentiles(raw_scores: dict[str, Decimal]) -> dict[str, Decimal]`
       - Converts raw scores to percentile ranks (0-100)
       - Percentile = rank / (n-1) * 100 (rank 0 = worst, rank n-1 = best)
       - Single trader: percentile = Decimal("100")
       - Empty input: returns {}
       - Handles ties: traders with same raw score get same percentile

    **Test cases (RED phase):**

    Recency weight tests:
    - Same day: returns Decimal("1.0")
    - Future timestamp: returns Decimal("1.0")
    - 90 days ago (half-life): result ~0.5 (within 0.01 tolerance)
    - 180 days ago: result ~0.25 (within 0.01 tolerance)
    - 0 days ago but different time: returns Decimal("1.0")

    Sample size confidence tests:
    - Below minimum (3 markets): returns Decimal("0")
    - At minimum (5 markets): returns positive value < 1
    - At full confidence (30 markets): returns Decimal("1.0")
    - Above full confidence (50 markets): returns Decimal("1.0")
    - Monotonically increasing between 5 and 30

    Expertise score tests:
    - Below minimum sample size (4 resolved): returns None
    - Exactly 5 resolved: returns ExpertiseScoreResult with valid score
    - High win rate + high concentration + recent activity: high raw_score
    - Low win rate + low concentration + old activity: low raw_score
    - Consistent trader (score=85, signal="stable"): consistency_multiplier = 1.05 (bonus)
    - Streaky trader (score=35, signal="streaky"): consistency_multiplier = 1.0 (no penalty, baseline)
    - Neutral consistency (score=60, signal="stable"): consistency_multiplier = 1.0 (baseline)
    - Insufficient data trader (signal="insufficient_data"): consistency_multiplier = 1.0 (baseline)
    - Custom weights override DEFAULT_WEIGHTS
    - Raw score clamped to [0, 100]
    - Void positions excluded from resolved count
    - percentile_rank is None (not yet computed)

    Percentile normalization tests:
    - Empty input: returns {}
    - Single trader: percentile = 100
    - Two traders: lower score = 0, higher score = 100
    - Three traders: 0, 50, 100
    - Tied scores get same percentile
    - All scores identical: all get same percentile
